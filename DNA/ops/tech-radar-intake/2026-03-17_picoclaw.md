# Tech Radar Intake - PicoClaw

Tool: PicoClaw
Discovered: 2026-03-17
Source: Sipeed GitHub repository
Link: https://github.com/sipeed/picoclaw
Status: Assess

## Problem It Solves

Provides a very lightweight local AI agent runtime for low-resource machines,
fast startup, and portable worker-style execution.

## Summary

- PicoClaw is positioned as a tiny local agent runtime with a web UI, CLI,
  multi-provider model support, and a strong emphasis on low memory use.
- The README highlights support for OpenRouter, Anthropic, OpenAI, Gemini,
  Groq, Ollama, and other providers, which makes it flexible as a worker-side
  runtime rather than a single-model tool.
- It also ships its own project memory and agent workflow surfaces such as
  `AGENTS.md`, `MEMORY.md`, `HEARTBEAT.md`, and scheduled jobs.
- The most important caution is explicit in the repo: the project is still in
  early development, may contain unresolved network-security issues, and is
  not recommended for production use before `v1.0`.

## Workspace Fit

- The best fit is the worker tier, not the orchestrator tier.
- It is interesting because the current stack already treats workers as
  interchangeable by cost and capability, so a tiny runtime could be useful
  for cheap local grunt tasks.
- It overlaps strongly with `OpenClaw`, which is the adopted gateway island in
  `STACK.md`, so this should not be framed as a replacement without formal
  stack-governance changes.
- Its own memory-file conventions are also a potential source of boundary
  confusion beside the DNA chain and the existing OpenClaw surface.

## Hot Take

Promising as a lightweight experimental runtime, but too early and too close
to the current gateway surface to advance beyond `ASSESS` right now.

## Next Step

Keep it on the radar as an `ASSESS` item for future low-resource worker
experiments. Revisit only if we want a tiny sidecar runtime for cheap local
execution and can keep the trial isolated from the main OpenClaw path.
