# System Health Monitor

This project monitors the workstation from both sides of the Windows/WSL boundary.

## Goal

Provide a reliable answer to:

- Is the Windows host healthy?
- Is the WSL guest healthy?
- Is the system configured correctly for sustained local AI and Docker workloads?
- What changed when performance or memory pressure spiked?

## Structure

- `/_scripts/system-health/collect-wsl.sh`
  Linux-side collector for WSL guest metrics.
- `/_scripts/system-health/collect-windows.ps1`
  Windows-side collector for host metrics.
- `/_scripts/system-health/summarize-health.py`
  Merges the latest host and guest snapshots into one summary view.
- `/_scripts/system-health/install-cron.sh`
  Installs or updates the WSL-side schedule.
- `/_scripts/system-health/install-windows-task.ps1`
  Copies the Windows collector to a Windows-local path and registers a Task Scheduler job.
- `/_scripts/system-health/build-windows-standalone-installer.sh`
  Generates a standalone Windows installer for Cloud PC or any other Windows host.
- `/_scripts/system-health/health-check.sh`
  Refreshes local snapshots when possible and prints a one-line machine status.
- `/_scripts/system-health/browser-drain-check.sh`
  Captures a browser-family memory baseline, waits, and reports how much RAM/commit came back after closing the apps.
- `/_scripts/system-health/storage-check.sh`
  Checks whether current or proposed paths align with the `C:` vs `S:` storage policy.
- `/_logs/system-health/`
  Structured output and latest snapshots.
- `/_logs/system-health/import/latest/`
  Optional import path for snapshots copied back from external Windows hosts such as a Cloud PC.

Windows-side local output defaults to:

- `C:\evo\system-health\logs\`

That keeps Windows collection independent of the WSL filesystem boundary. The WSL summary step reads those Windows snapshots through `/mnt/c/evo/system-health/logs/`.

## Design Rules

- Do not infer Windows facts from WSL mount paths.
- Do not infer WSL pressure from Windows-only process views.
- Store machine-readable output first, human-readable summaries second.
- Prefer timestamped JSON records over plain text logs.
- Keep collectors independent so one side can fail without blocking the other.
- Tag every snapshot with a machine key so the same collector can run on EVO-STATION, a Cloud PC, or another Windows host without overwriting data.

## Expected Outputs

Collectors will write JSON Lines records such as:

- `/_logs/system-health/wsl.jsonl`
- `/_logs/system-health/windows.jsonl`

The summary step will maintain:

- `/_logs/system-health/latest.json`
- `/_logs/system-health/latest.txt`
- `/_logs/system-health/health-alerts.txt`
- `/_logs/system-health/trends.json`
- `/_logs/system-health/trends.txt`
- `/_logs/system-health/notification-state.json`

Quick terminal check:

- `/_scripts/health-check.sh`
- `/_scripts/system-health/health-check.sh`
- `/_scripts/browser-drain-check.sh`
- `/_scripts/storage-check.sh`

## Initial Scope

Windows collector:

- Physical RAM usage
- Commit usage
- `VmmemWSL` memory
- `C:` and `S:` drive free space
- WSL VHDX path and size
- `.wslconfig` presence and relevant values
- Top memory consumers

WSL collector:

- `MemAvailable`
- Swap usage
- Filesystem usage for `/` and workspace paths
- Docker/container counts and normalized memory totals
- Key service status
- Top memory consumers
- Optional PSI pressure metrics where available

## Scheduling

WSL side:

- `cron` drives the Linux collector

Windows side:

- Task Scheduler drives the PowerShell collector

Recommended initial frequency is every 15 minutes. Move to 5 minutes only if you need tighter spike correlation.

## Multi-host Note

The Windows collector is host-local. If you want visibility into a Windows 365 Cloud PC, run the same `collect-windows.ps1` script there as well. The script tags its output with the Windows hostname, so EVO-STATION and the Cloud PC remain separate hosts in the model.

For practical deployment on a separate Windows host, use the standalone installer generated by `build-windows-standalone-installer.sh`. Deployment steps are documented in `DEPLOY_WINDOWS.md`.

Important boundary: a remote Windows VM is not captured just because you access it from your local Windows desktop. The local Windows collector can only observe EVO-STATION. To get real VM-side analytics, the Windows collector must run inside that VM or its snapshots must be imported back into this workspace.

## Alerting

When the merged machine status escalates from `ok` to `warn` or `critical`, the summary step attempts a Windows desktop notification and records the last-seen status in `notification-state.json`. Alerts also carry remediation suggestions in the text outputs.
